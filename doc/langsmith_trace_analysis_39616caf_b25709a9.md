# 初始生成与再次编辑失效 Trace 分析

分析对象：

- 初始生成：`39616caf-ed8b-4e4b-985e-4ed4f5d0ff83`
- 运行时错误编辑：`b25709a9-0165-4af4-a99e-35d5261a665c`
- 对比代码：当前 `v4` 与本地 `v3`

## 结论

编辑模型已经生成有效修复，但 v4 的编辑验收错误地要求诊断模型选中的 `initScene` 必须发生文本变化。实际根因位于 `renderFormula`：该函数把调用方传入的 `HTMLSpanElement` 再传给 `document.querySelector`。整页编辑候选已经把 `renderFormula` 改为同时接受选择器字符串和 DOM 元素，但最终被错误验收回滚。

再次编辑的产品设计本身有效。前端以完整 HTML 快照创建不可变子分支，并将新分支设为活动页和下一次编辑的 `current_html`；无需增加后端 HTML 持久化、diff 合并或会话内隐式状态。失败发生在服务端接受候选之前，因此前端只能保留原分支。

## 完整链路

### 初始生成

1. 首稿模型输出 29,470 字符，装配后 41,379 字符，耗时 145.55 秒。
2. 首次校验发现 3 个 `inline_event` 和 1 个 `animation_controller_bypass`；另有 `gsap_mutates_serialized_state`、`svg_visual_center_mismatch` warning。
3. 确定性修复把业务 HTML 调整到 29,988 字符，清除了内联事件，但仍有 `animation_controller_bypass`。
4. 模型修复耗时 85.22 秒，输出 29,866 字符；静态校验通过后交付。
5. 原校验器只检查 JS 语法、结构和既有运行时契约，没有识别“辅助函数参数作为 selector，但调用点传入 Element”的跨函数类型错误，因此首稿仍会在初始化公式时抛错。

### 编辑失败

1. 编辑输入为装配后 41,774 字符、业务 HTML 25,047 字符，包含明确的 `querySelector`/`HTMLSpanElement` 运行时错误。
2. 编辑诊断耗时 8.03 秒，以 0.95 置信度把 `initScene` 认作唯一目标；这是无调用栈条件下的错误推断。
3. 函数补丁模型耗时 7.06 秒，原样返回 `initScene`，哈希保护正确地拒绝了无变化补丁。
4. 工作流随后回退整页编辑，耗时 135.62 秒，输出 25,161 字符。候选将 `renderFormula(selector, latex)` 改为按参数类型分流，实际已经解决运行时错误。
5. 通用候选验收仍绑定原诊断目标，因 `initScene` 未变化返回 `edit_function_not_changed:initScene`，导致整个有效候选回滚。

## v3 对比

v3 的编辑链路只有“当前完整 HTML + 修改意见 → 整页模型编辑 → 解析/校验/修复”，不做诊断函数哈希验收，所以这类修改通常能够生效。它的优点是路径短、不依赖诊断准确率；缺点是每次都整页重生成，耗时和 Token 成本高，也更容易产生无关改动。

v4 的分层编辑方向是合理的，问题是把“诊断目标函数变化”误当成“用户意图满足”的必要条件。诊断只能用于缩小候选范围，不能成为唯一事实；运行时错误、静态契约和最终候选差异才应决定是否接受。

## 已实施修复

- 新增通用 DOM API 参数契约检查：检测辅助函数调用 `querySelector(param)`，但已证明调用点传入 DOM 元素的情况。
- 将该错误加入首稿硬校验和确定性修复，统一把辅助函数改为“字符串时查询、Element 时直接使用”，避免进入昂贵的模型修复。
- 编辑请求携带对应运行时错误时，优先执行确定性函数补丁，跳过诊断模型、函数模型和整页模型。
- 运行时编辑验收不再强制诊断目标函数必须改变；若实际根因位于其他函数，只要业务函数确有变化并且确定性错误契约通过，就接受候选。
- 保持前端现有不可变 HTML 分支与完整快照策略，不增加新的前后端接口或持久化状态。

## 剩余边界

静态检测仅阻断具有高置信度数据流证据的 DOM 参数类型错误，以避免正则推断造成误报。无法从源码静态证明的浏览器运行时问题仍依赖用户上报的错误、函数补丁或整页编辑；完整的动态行为验证继续属于离线浏览器回归，不进入生产同步链路。
